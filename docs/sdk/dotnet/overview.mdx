The AG-UI .NET SDK connects the AG-UI protocol to `Microsoft.Extensions.AI`. On the server, it adapts an AG-UI `RunAgentInput` into `ChatMessage` and `ChatOptions`, calls any `IChatClient`, and streams the response back as AG-UI events. On the client, it wraps an AG-UI endpoint as an `IChatClient`, so .NET applications can consume an AG-UI agent through the standard MEAI middleware pipeline.

## Packages

The current SDK PR adds three packages under `sdks/dotnet/src`.

| Package | Use it when you want to | What it contains |
| --- | --- | --- |
| `AGUI.Abstractions` | Share AG-UI protocol types across clients, servers, tests, and framework adapters. | Events, messages, tools, capabilities, interrupts, `RunAgentInput`, raw event preservation, and `AGUIJsonSerializerContext`. |
| `AGUI.Hosting.AspNetCore` | Expose an ASP.NET Core endpoint that speaks AG-UI. | `AddAGUI()`, `ToChatRequestContext(...)`, `AsAGUIEventStreamAsync(...)`, `AGUIStreamOptions`, and hosting conversion helpers. |
| `AGUI.Client` | Call an AG-UI endpoint from .NET. | `AGUIChatClient`, `IAGUITransport`, and the HTTP/SSE transport that converts AG-UI events into `ChatResponseUpdate` values. |

When preview packages are published, the expected install shape is:

```bash
dotnet add package AGUI.Abstractions
dotnet add package AGUI.Hosting.AspNetCore
dotnet add package AGUI.Client
```

Confirm package IDs and versions against the final NuGet release before publishing the install commands.

## How the bridge works

AG-UI and MEAI represent the same conversation at different layers.

On the hosting side:

1. An AG-UI client posts a `RunAgentInput`.
2. `ToChatRequestContext(...)` converts AG-UI messages and tools into `ChatMessage` and `ChatOptions`.
3. Your `IChatClient` streams `ChatResponseUpdate` values.
4. `AsAGUIEventStreamAsync(...)` converts those updates into AG-UI lifecycle, message, tool, state, reasoning, raw, custom, and interrupt events.
5. ASP.NET Core writes the events as Server-Sent Events.

On the client side:

1. Your .NET app creates an `AGUIChatClient`.
2. The client builds a `RunAgentInput` from `ChatMessage`, `ChatOptions`, tools, state, and resume data.
3. The HTTP transport posts to the AG-UI endpoint and reads the SSE event stream.
4. The event converter produces `ChatResponseUpdate` values for your .NET app and MEAI middleware.

## Compatibility model

The SDK maps common protocol concepts into MEAI-native shapes:

- Text events become streamed text content.
- Tool calls and tool results become `FunctionCallContent` and `FunctionResultContent`.
- Client tools flow through MEAI function invocation and AG-UI interrupt/resume semantics.
- State, reasoning, custom, and framework-specific content can be emitted through `AGUIStreamOptions`.
- AG-UI events that do not map cleanly to MEAI stay available through raw representations instead of being discarded.

Microsoft Agent Framework consumes this bridge. The shared SDK docs cover the AG-UI and MEAI mapping layer. Agent Framework-specific workflow, step, and agent events belong in Agent Framework docs and tests.
