You have an AG-UI endpoint and want to call it from .NET code. The client package should make that endpoint look like a standard `IChatClient`, so existing `Microsoft.Extensions.AI` middleware and application patterns continue to work.

## Reader journey

Start with a remote AG-UI endpoint URL. End with an `AGUIChatClient` that streams `ChatResponseUpdate` values and composes with MEAI middleware for function invocation, logging, telemetry, retry, caching, and structured output.

## Core concepts to explain

### Create the client

Docs should show the final constructor or builder shape for connecting an `HttpClient` to an AG-UI base URL.

Final examples need to verify:

- Required endpoint URL shape.
- Authentication and header configuration.
- Cancellation behavior.
- How connection errors surface through MEAI abstractions.

### Send messages and stream updates

The client should accept ordinary `ChatMessage` values and return ordinary `ChatResponseUpdate` values.

Docs should cover:

- Text content streaming.
- Tool call content and tool result content.
- Error content.
- Raw representations for AG-UI events that do not map cleanly to MEAI content.
- How to accumulate streamed updates into a multi-turn conversation.

### Keep MEAI middleware composable

Because `AGUIChatClient` is an `IChatClient`, readers should be able to use the same middleware they use with direct model clients.

Docs should cover:

- Local tool execution through function invocation middleware.
- Telemetry and logging around a remote AG-UI call.
- Retry and caching boundaries for streaming calls.
- Structured output considerations when the server emits protocol events.

## Example sequence to finalize

The final page should include one complete console sample that:

1. Creates an `HttpClient`.
2. Creates an `AGUIChatClient`.
3. Sends a user `ChatMessage`.
4. Streams response updates.
5. Handles text, tool call, tool result, and error content.
6. Preserves messages for a follow-up turn.
